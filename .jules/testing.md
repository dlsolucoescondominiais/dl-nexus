## 2024-05-02 - Mocking Internal Methods and Fixing Key Mismatches
**Learning:** When writing unit tests for code that wasn't thoroughly reviewed by the initial author, there can be discrepancies between documentation or issues and the actual code. Always verify the variable names (like `mensagem` vs `mensagem_original`) inside the target function before mocking the input data. Also, when an issue specifically requests to mock a method (e.g. `calcular_porte`), do not assume it's trivial enough to leave unmocked; explicit instruction requires explicit mocking to ensure test isolation.
**Action:** Always read the target function implementation directly, verifying dictionary key names, before writing the test inputs. Ensure all requested dependencies are mocked, regardless of their complexity.
